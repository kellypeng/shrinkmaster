## Two surfaces, different deploy models — DO NOT confuse them

ShrinkMaster has **two distinct deploy surfaces** with very different behaviors. Most "why isn't my change live" bugs come from confusing them.

### Surface 1 — the landing page (`docs/`)

- Source of truth: `docs/index.html` and the rest of `docs/`
- Deploy: `git push` → GitHub Pages auto-rebuilds in ~1 minute → live at `shrinkmaster.kellypeng.com`
- Updates are immediate. No manual step.
- Anything in `docs/`, `README.md`, `integrations/`, top-level project files: web-only.

### Surface 2 — the macOS app (`.dmg` on GitHub Releases)

- Source of truth: the actual `.dmg` file uploaded to a GitHub Release
- Deploy: must run `./scripts/release.sh` to rebuild + re-upload
- **`git push` does NOT update what users download.** The `.dmg` is a frozen snapshot.
- Until you rebuild and re-upload, downloaders keep getting the previous build no matter how many code changes you push.

### Things that live INSIDE the `.app` (changing them requires a rebuild)

If you change any of these, run `./scripts/release.sh` to ship a new build:

- `build/icon.icns` — the app icon (regenerated by `./scripts/build-icon.sh` from `build/icon.svg`)
- `bin/ffmpeg` and `bin/ffprobe` — bundled binaries
- `main.js`, `preload.js`, `renderer.js`, `index.html` — the app source
- `build/entitlements.mac.plist`
- `package.json` fields: `version`, `productName`, `appId`, `mac.*`, `extraResources`

### How to know which surface a change affects

Ask: "If a user downloaded the `.dmg` yesterday, does my change affect what they see when they launch the app?"

- **Yes** → rebuild required (run `./scripts/release.sh`)
- **No** (web-only) → just `git push`

When in doubt: assume rebuild is required.

## Release flow

For a patch to the **existing** version (just rebuilding with new content, same version):

```bash
./scripts/release.sh
```

The script: kills running app instances → regenerates icon → cleans `dist/` → builds DMG → copies to stable + version-named filenames → uploads with `--clobber` to the existing release.

For a **new version** (bump version):

1. Bump `version` in `package.json` (e.g. `0.1.0` → `0.2.0`)
2. Commit and tag: `git tag v0.2.0 -m "..."` and `git push origin v0.2.0`
3. Create the release: `gh release create v0.2.0 --title "..." --notes "..."`
4. Run `./scripts/release.sh` to build and upload to the new release

## Download URL patterns

Two URL patterns exist on every release for backward compatibility:

- **`ShrinkMaster-mac-universal.dmg`** — stable name. Landing page links here. Recommended for sharing.
- **`ShrinkMaster-0.1.0-universal.dmg`** — version-named. Anyone who bookmarked this URL before May 2026 still works.

The stable name comes from `package.json` `mac.artifactName`. `scripts/release.sh` makes the version-named copy by hand for backward compat.

The landing page uses `https://github.com/kellypeng/shrinkmaster/releases/latest/download/ShrinkMaster-mac-universal.dmg` — GitHub auto-redirects to whichever release is latest, so the URL never needs updating across versions.

## Bilingual UI

Every user-visible string in `renderer.js` (the app) and `docs/index.html` (the landing) lives in an `I18N` dictionary defined at the top of each file. Both **EN and 中文** entries are required when adding a new string. The default text inside the HTML element is the EN fallback for the no-script case.

When adding a string:
1. Add a new entry to the `I18N` object — must include `en` and `zh` keys
2. Reference it via `data-i18n="key"` (textContent) or `data-i18n-placeholder="key"` (input placeholder)
3. Provide a sensible EN default as the element's initial text content

## Voice rules for any text the user sees

- **No em dashes (—) in body copy.** Use periods, commas, or rewrite. Em dashes have become the visual tell of LLM-written copy.
- **No "we"** if it's first-person. The project is solo; say "I" or leave the actor out.
- **No corporate adjectives** ("seamless / delightful / robust / blazing"). Be specific.
- **No fake AI claims**, fake user counts, fake testimonials.
- **No `print()`/`console.log()`** lecturing about why open source / privacy / etc. is good. Show, don't preach.

These apply to: README, landing page copy, app UI strings, release notes, commit messages destined for users (release notes pulled from commits).
